# write-post

这个 skill 帮你在博客 / 文档站点中撰写技术文章，支持任意文档站点生成器。

## 第一步：探测项目约定

在动笔之前，先读取项目配置，确定以下三件事：

1. **内容目录**（文章放在哪里）
2. **Frontmatter 结构**（必填字段、可选字段、格式）
3. **文件命名规则**（slug 格式、扩展名）

### 探测顺序

按以下顺序查找项目配置，**找到即止**：

| 优先级 | 文件 / 特征 | 生成器 | 默认内容目录 |
|--------|-------------|--------|--------------|
| 1 | `blog.config.ts` / `blog.config.js` | 自定义博客框架 | 读取配置中的 `postsDir` 或 `contentDir` 字段 |
| 2 | `astro.config.*` + `src/content/` | Astro | `src/content/blog/` 或 `src/content/posts/` |
| 3 | `next.config.*` + `posts/` 或 `content/` | Next.js (MDX/静态) | `posts/` 或 `content/` |
| 4 | `vitepress` 字段 / `.vitepress/config.*` | VitePress | `docs/` 或 `guide/` |
| 5 | `docusaurus.config.*` | Docusaurus | `docs/` 或 `blog/` |
| 6 | `hugo.toml` / `hugo.yaml` / `config.toml` | Hugo | `content/posts/` |
| 7 | `_config.yml` / `_config.toml` | Jekyll | `_posts/` |
| 8 | `mkdocs.yml` | MkDocs | `docs/` |
| 9 | `nuxt.config.*` + `@nuxt/content` | Nuxt Content | `content/` |
| 10 | 根目录存在 `posts/` 目录 | 未知 | `posts/` |
| 11 | 根目录存在 `content/` 目录 | 未知 | `content/` |
| 12 | 兜底 | 未知 | `posts/`（并提示用户确认） |

**探测方式：**
- 用 `ls` 或 `find` 检查项目根目录的配置文件是否存在
- 读取配置文件提取内容目录路径
- 查看已有文章的 frontmatter 作为格式参考（`ls <内容目录>` → 读取 1–2 篇已有文章）

### 从已有文章推断 Frontmatter

读取 1–2 篇已有文章后，提取其 frontmatter 字段作为模板，例如：

```yaml
# 示例：从已有文章中观察到的结构
---
title: ...
date: ...        # 或 pubDate / publishDate / created
tags: [...]      # 或 categories / topics
description: ... # 或 summary / excerpt
draft: false     # 若存在则保留
---
```

如果项目中没有已有文章，则使用以下最小化通用 frontmatter：

```yaml
---
title: 文章标题
date: YYYY-MM-DD
tags:
  - 标签
---
```

## 写作风格指南

风格根据文章主题灵活切换：

- **技术教程型**（如"如何实现 X"）：步骤清晰，配代码示例，实操性强，语言简练直接
- **深度分析型**（如"为什么选择 X"、"X vs Y"）：有自己的观点，讲清楚背后的原因，可以有对比表格
- **随笔/思考型**（如经验总结、感悟）：语气自然随意，不需要严格结构，第一人称

**通用原则：**
- 中文为主，技术术语保留英文（如 useState、Promise、TypeScript）
- 开篇直接切入主题，不要废话铺垫
- 代码示例用三反引号 + 语言标识（` ```typescript `）
- 标题层级不超过三级（#、##、###）
- 结尾可以有总结，但不必强制

### 引用概念与文档

提到某个概念、规范、工具、官方文档时，用 Markdown 链接形式引用，而不是裸文本：

- 提到 [MDN](https://developer.mozilla.org)、[RFC 规范](https://www.rfc-editor.org)、[某库的文档](https://example.com)——给链接
- 提到某篇站内已有文章——用相对路径链接
- 提到某个 GitHub issue / PR / commit——链接到对应页面
- 只是泛泛提到一个概念名词（不是特定文档）——不需要强行加链接

目的是让读者能顺手跳过去，不用自己去搜。

### 写出人味：避免 AI 腔

文章要读起来像人写的，不是机器生成的。关键是语气真实、有立场、不圆滑。

**禁用的 AI 套话（直接删掉或改写）：**
- "需要注意的是……" → 直接说要注意什么
- "值得一提的是……" → 直接提
- "总的来说" / "综上所述" / "不难发现" → 删掉，换成真实结论
- "本文将……" / "我们将……介绍……" → 开头废话，删掉
- "希望本文对你有所帮助" → 删掉
- 每段都以"首先……其次……最后……"结构 → 太公式化，拆散它

**让文章有人味的写法：**
- 偶尔用口语连接词：不过、其实、说实话、说白了、反正、嗯……
- 可以有不完美的地方：没想清楚的说"我也没完全想清楚"，有争议的不强行给结论
- 观点要有立场，不要两边都说"各有优劣"然后不表态
- 允许轻微跑题：一个类比、一个题外话，反而更真实
- 句子长短要错落，不要每句话都差不多长
- 技术文章里可以插一句"这个坑我踩过，很疼"之类的个人经历
- 不要过度使用加粗，加粗泛滥等于没加粗

### 口语风格因文章类型而异

不同类型的文章，口语的「味道」不一样，不要用一套统一的"技术博主腔"套所有文章：

**技术教程型** — 口语偏「边做边说」：
> 先 `npm install`，装完之后你会发现多了一个 `.cache` 目录，不用管它。
> 这里有个小坑：如果你用 pnpm，路径会不一样，记得换一下。

**深度分析型** — 口语偏「在想这件事」：
> 我一直有个疑问：为什么大家都说 Rust 的错误处理好，但真正用起来又觉得很繁琐？
> 后来我想明白了——"好"和"顺手"是两件事。

**随笔/思考型** — 口语偏「跟朋友说话」：
> 说实话，那段时间挺难熬的。每天对着同一个报错，感觉自己在原地转圈。
> 后来有个同事顺口说了一句话，突然就想通了。

核心原则：口语要自然地**长**在文章里，不是硬插几个"其实"就算有人味了。写完一段，自己读一遍，如果读起来像念稿子，就说明还不够。

### 图例：用 SVG 而不是代码块

讲解架构关系、数据流向、层次结构、状态机、对比关系这类概念时，文字和代码块的表达力有限——ASCII 方框图勉强能看，但在移动端和不同字体下经常错位。用 SVG 来画。

#### 什么时候用 SVG

- 架构分层图（比如"三个 SDK 的关系"）
- 数据/请求流向图（A → B → C）
- 状态转换图、生命周期图
- 概念对比图（并排比较两个方案）
- 任何你本能想画个框、连个箭头的场景

#### 什么时候不用

- 能用一句话或一个列表说清楚的——别画蛇添足
- 纯代码逻辑——代码块就是最好的图
- 表格能搞定的对比——用 Markdown 表格

#### 存储方式：SVG 保存为独立文件

SVG 不要直接写在 Markdown 里——内联 SVG 会让 Markdown 源文件又长又难读。把 SVG 保存到项目的静态资源目录，在 Markdown 中用 `<img>` 引用。

存放路径规范：`public/images/<文章slug>/` 目录下，每个 SVG 一个文件，文件名用英文描述性短语。

```markdown
<!-- 在 Markdown 中这样引用 -->
<img src="/images/my-article/architecture.svg" alt="系统架构图" width="100%" style="max-width:560px" />
```

`alt` 属性要写有意义的描述，方便无障碍访问和 SEO。

#### 怎么画

几条原则：

1. 通过 `viewBox` 控制实际尺寸，不要设固定 `width`/`height`（由 `<img>` 标签控制）
2. 保持简洁：用圆角矩形（`rx="8"`）做节点，直线或折线做连接，箭头用 `marker-end`
3. 字号用 14-16px，确保在小屏幕上也能看清
4. 关键节点可以用不同的填充透明度区分层次，但不要超过 3-4 种

#### 暗色/亮色主题兼容（很重要）

博客有暗色和亮色两种主题，SVG 必须在两种模式下都看得清。

因为 SVG 是通过 `<img>` 引用的外部文件，无法继承页面的 `currentColor`，所以要在 SVG 内部用 `<style>` + `@media (prefers-color-scheme: dark)` 来切换颜色：

```xml
<svg viewBox="0 0 400 200" xmlns="http://www.w3.org/2000/svg">
  <style>
    .fg { fill: #1a1a1a; }
    .fg-sub { fill: #1a1a1a; opacity: 0.5; }
    .stroke { stroke: #1a1a1a; }
    .box { fill: rgba(0,0,0,0.06); stroke: rgba(0,0,0,0.15); }
    .arrow-head { fill: #1a1a1a; }
    @media (prefers-color-scheme: dark) {
      .fg { fill: #e5e5e5; }
      .fg-sub { fill: #e5e5e5; opacity: 0.5; }
      .stroke { stroke: #e5e5e5; }
      .box { fill: rgba(255,255,255,0.08); stroke: rgba(255,255,255,0.2); }
      .arrow-head { fill: #e5e5e5; }
    }
  </style>
  <!-- 用 class 而不是内联样式 -->
  <rect x="50" y="10" width="300" height="50" rx="8" class="box" stroke-width="1.5"/>
  <text x="200" y="40" text-anchor="middle" class="fg" font-size="14">节点名称</text>
</svg>
```

核心思路：
- 亮色模式：深色文字（`#1a1a1a`）+ 浅灰背景（`rgba(0,0,0,0.06)`）
- 暗色模式：浅色文字（`#e5e5e5`）+ 深灰背景（`rgba(255,255,255,0.08)`）
- 如果需要区分不同节点，用不同的 `rgba` 透明度值（0.04、0.08、0.12）来做层次感
- 用 CSS class（`.fg`、`.box`、`.stroke`）统一管理，不要在每个元素上写内联 fill/stroke

## 技术内容查证：调用 find-docs

当文章涉及具体的库、框架、SDK、CLI 工具或技术概念时，不要凭训练数据里的印象去写——API 会变、配置会改、版本会升级，写出来的东西很可能已经过时了。

**写之前先查文档。** 用 `/find-docs` skill（Context7 CLI）拉取最新的官方文档和代码示例，然后基于查到的内容来写。

### 什么时候需要查

- 文章要展示某个库的 API 用法、配置项、CLI 命令
- 要对比不同技术方案的特性或写法
- 涉及某个工具的安装、初始化、迁移步骤
- 提到的技术细节你不是 100% 确定还是最新的

### 什么时候不用查

- 纯经验分享、感悟、随笔类文章，不涉及具体 API
- 只是泛泛提到某个技术名词，不展开讲用法
- 用户明确说了"不用查文档"或"按你的理解写"

### 怎么查

用 find-docs skill 的两步流程：

```bash
# 1. 解析库 ID
ctx7 library <库名> "<你要查的具体问题>"

# 2. 拉取文档
ctx7 docs <库ID> "<具体问题>"
```

如果文章涉及多个技术栈，每个都要查。查完之后把关键信息（正确的 API 签名、配置格式、注意事项）融入文章，不要原封不动搬运文档——读者看的是你的文章，不是官方文档的复读。

### 查到的信息怎么用

- 代码示例要基于最新文档来写，不要用记忆里可能过时的写法
- 如果查到的内容和你印象里的不一样，以文档为准
- 在文章中用链接指向官方文档，方便读者深入阅读
- 如果某个 API 有版本差异，文章里要说清楚适用的版本

## 工作流程

### 模式 1：起草完整文章

收到话题或简单描述后：

1. 探测项目约定（内容目录 + frontmatter 结构）
2. **如果文章涉及具体技术，先用 find-docs 查证最新文档**
3. 根据话题判断合适的写作风格
4. 选取合适的 tags（2–4 个）
5. 基于查到的文档 + 自身理解，生成完整文章（含 frontmatter）
6. 询问用户是否写入文件（或用户一开始就要求写入时直接写）

### 模式 2：文章大纲规划

如果用户想先规划结构：列出各节标题 + 每节一句话说明要点，等用户确认后再展开写。

### 模式 3：扩展 / 润色现有内容

用户提供草稿片段时：
- **扩展**：补充细节、添加代码示例、丰富说明
- **润色**：保持原意，改善表达流畅性和逻辑性，不要过度改写

### 模式 4：批量处理多篇文章

当任务涉及多篇文章（更新系列、重写多个文件、对多篇文章套用同一改动）时：

1. 列出所有待处理的文章路径，确认范围
2. **为每篇文章各启动一个 sub-agent 并行处理**，每个 sub-agent 独立完成：探测约定 → 起草/修改 → 写入文件
3. 等全部 sub-agent 完成后，进入评审阶段（见下方）

---

## 评审阶段（所有模式通用）

无论处理的是一篇还是多篇文章，写入完成后都要跑一轮评审：

**启动一个独立的 reviewer sub-agent**，给它传入所有刚写完的文章路径，让它从读者视角逐篇评审，重点检查：

- 写作风格是否符合文章类型（教程/分析/随笔）
- 口语味道是否自然，有没有 AI 腔残留
- 引用概念时有没有该加链接但没加的地方
- 逻辑结构是否清晰，有没有跳跃或重复
- 有没有用 ASCII 代码块画图的地方应该换成 SVG
- frontmatter 信息是否准确完整

reviewer 输出一份简明的改进意见清单（按文章分组，每条意见说明问题在哪、建议怎么改）。

收到评审意见后，根据具体建议逐项修改文章。如果某条意见涉及多篇文章中的共性问题（比如都有 AI 腔、都缺链接），可以再次并行用 sub-agent 处理。

修改完成后告知用户，同时附上一句话的改动摘要（改了哪些方面）。

## 写入文件

写入路径：`<探测到的内容目录>/<slug>.md`（或 `.mdx`，根据项目已有文件扩展名决定）

slug 从标题推导：中文标题转为英文描述性短语，英文标题直接转 kebab-case。

如果探测到的内容目录不确定（走了兜底逻辑），写入前先告知用户探测结果并请求确认。

## 日期

使用今天的实际日期作为 frontmatter 中的 date（格式 YYYY-MM-DD）。
