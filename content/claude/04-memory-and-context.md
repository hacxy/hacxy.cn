# 第四章：Memory 与持久化上下文

Claude Code 每次新对话都是从零开始的——没有记忆，不知道你上次说了什么，也不知道这个项目的背景。持久化上下文系统就是为了解决这个问题：把"需要一直记着的事情"固化下来，让每次对话都能带着正确的背景开始。

Claude Code 有三种主要的持久化机制，分工不同，配合使用效果最好。不过在讲持久化之前，先搞清楚一件事：每次对话里的"临时上下文"是怎么管理的。毕竟持久化解决的是"跨对话"的问题，而"单次对话内"的上下文管理同样关键。

## 会话上下文管理

Claude Code 有一个上下文窗口（context window），你可以把它想象成 Claude 的"工作记忆"——所有对话内容、读取的文件、命令输出、CLAUDE.md 内容都放在这个窗口里。它的容量是有限的，用满了就得压缩。

### 上下文里装了什么

每次启动会话，Claude Code 会按顺序加载这些内容：

1. **系统提示词**（你看不到，大约 4200 token）
2. **Auto Memory**（MEMORY.md 的前 200 行或 25KB）
3. **环境信息**（工作目录、平台、git 分支等）
4. **MCP 工具名称**（只加载名字，不加载完整定义）
5. **Skill 描述**（一句话描述，用到时才加载全文）
6. **CLAUDE.md 文件**（全局 → 项目 → 本地，按层级加载）
7. **你的提问**
8. **后续的文件读取、命令输出、Claude 的回复……**

你会发现，还没开始干活，上下文窗口里就已经有不少东西了。后续每读一个文件、每跑一个命令，输出都会占用上下文空间。

### 上下文快满了会怎样

Claude Code 会在上下文使用到 95% 时自动压缩（auto-compaction）。压缩的意思是：把之前的对话历史总结成一段摘要，替换掉原始内容。你的请求和关键代码片段会保留，但对话早期的详细指令可能会丢失。

这里有个隐蔽的坑：**上下文退化**（context rot）。不是说压缩了就完事了——随着对话变长，模型的注意力分散到更多 token 上，决策质量会悄悄下降。常见的症状是 Claude 开始重复之前试过但失败的方案，或者引用已经过时的文件状态。

### 手动管理上下文

有两个命令用来主动管理：

**`/compact [指令]`** —— 压缩但不清除。把对话历史替换成摘要，可以带上指令告诉它保留什么重点：

```
/compact 重点保留 API 模块的改动和当前的测试失败信息
```

压缩后，项目根目录的 CLAUDE.md 会从磁盘重新读取并注入上下文，所以 CLAUDE.md 里的规则不会丢。但子目录的 CLAUDE.md 不会自动重新加载——得等 Claude 下次读取那个目录的文件时才会触发。

**`/clear`** —— 彻底清空，重新开始。对话历史全部丢弃，只保留启动时自动加载的内容（CLAUDE.md、Memory 等）。清空前的会话会被保存，之后可以用 `/resume` 找回来。

**`/context`** —— 查看当前上下文的使用情况，用一个可视化的网格展示哪些内容占了多少空间。想知道为什么上下文快满了，跑一下就知道。

### 什么时候该 compact，什么时候该 clear

这个选择其实很简单：

- **任务还没做完，但对话已经很长了** → `/compact`。保留上下文的摘要，继续当前的工作。带上指令可以引导压缩时保留什么。
- **一个任务做完了，要开始下一个** → `/clear`。干净地重新开始，不带上一个任务的噪音。
- **发现 Claude 开始犯迷糊**（重复尝试失败的方案、引用错误的文件状态）→ `/clear`。这时候 compact 可能救不回来，因为问题出在"记了太多没用的东西"。

**最佳策略：一个任务一次对话。** 任务之间用 `/clear` 隔开。对话越短越聚焦，Claude 的表现越好。如果一个任务本身很大，就用 `/compact` 在中间手动压缩，把焦点放在当前阶段需要的信息上。

## 跨会话延续

有时候一个任务没做完就得离开，或者第二天想接着昨天的工作继续。Claude Code 的会话会自动保存到磁盘（`~/.claude/projects/` 下的 JSONL 文件），所以你随时可以回来。

### 继续上一次会话

最常用的两个命令：

```bash
# 继续最近一次会话（最常用）
claude -c

# 打开会话选择器，从列表里选一个
claude -r
```

`claude -c` 非常方便——关掉终端、切了个分支、处理了个紧急事情，回来直接 `claude -c` 就能接上。

在已有的对话里，也可以用 `/resume` 切换到另一个会话。

### 给会话起名字

如果你同时在做好几件事，默认的会话列表就是一堆时间戳，根本分不清哪个是哪个。给会话起个名字：

```bash
# 启动时起名
claude -n auth-refactor

# 对话中起名
/rename auth-refactor
```

起了名字之后，可以直接用名字恢复：

```bash
claude --resume auth-refactor
```

### 会话分支

有时候你想试一个不同的方案，但又不想丢掉当前的进度。用 `/branch` 创建一个会话分支：

```
/branch try-streaming-approach
```

分支会复制当前的完整对话历史到一个新会话，原来的会话保持不变。试完了如果不好，随时可以切回原来的会话。

从命令行也可以分支：

```bash
claude --continue --fork-session
```

### 会话选择器

跑 `claude -r` 或在对话中输入 `/resume` 会打开一个交互式的会话选择器，支持这些快捷键：

| 快捷键 | 功能 |
|--------|------|
| `↑` / `↓` | 上下浏览会话 |
| `Enter` | 恢复选中的会话 |
| `Space` | 预览会话内容 |
| `Ctrl+R` | 重命名选中的会话 |
| `Ctrl+B` | 只显示当前 git 分支的会话 |
| `Ctrl+W` | 显示同一仓库所有 worktree 的会话 |
| `Ctrl+A` | 显示这台机器上所有项目的会话 |

每行会展示会话名称（或第一条消息的摘要）、最后活动时间、消息数量和 git 分支。

### 什么时候继续、什么时候新开

- **继续（`claude -c`）：** 任务中途被打断、需要接着做。上下文还在，不用重新解释背景。
- **新开：** 开始一个新任务。新的上下文窗口更干净，Claude 的注意力更集中。

一个经验法则：如果你需要花超过两句话解释"上次做到哪了"，那大概率还不如 `claude -c` 直接继续省事。如果上次的任务已经做完了，就果断新开。

## CLAUDE.md：你写给 Claude 的规则

CLAUDE.md 是你主动维护的规则文件，内容是"项目约定"和"行为要求"。[上一章](./03-core-features)介绍过基础用法，这里补充一些进阶技巧。

### 保持简洁

CLAUDE.md 建议控制在 200 行以内。越长，Claude Code 遵守的准确率越低。模型在一个上下文窗口里大概能可靠地遵循 150-200 条独立指令，而 Claude Code 自己的系统提示词已经占了大约 50 条的"名额"——留给你的 CLAUDE.md 的有效空间其实没那么多。

**好的 CLAUDE.md 内容：**
- 禁止做什么（"不允许用 `any`"、"不要自动 commit"）
- 项目特有的约定（"API 错误统一用 `AppError` 类"）
- 不显而易见的命令（"跑测试要用 `pnpm test:unit`，不是 `pnpm test`"）

**不适合放的内容：**
- 能从代码里直接看出来的架构
- 通用的编程最佳实践（Claude 本来就知道）
- 长篇的背景介绍
- 多步骤的工作流程（放 Skill 更合适）

### 加载顺序

CLAUDE.md 不是只有一个文件——它可以存在于多个位置，Claude Code 会按从上到下的优先级全部加载并拼接：

| 层级 | 位置 | 用途 |
|------|------|------|
| 组织策略（最高优先级） | macOS: `/Library/Application Support/ClaudeCode/CLAUDE.md` | IT/运维部署的全公司规则，不可被覆盖 |
| 用户全局 | `~/.claude/CLAUDE.md` | 你的个人偏好，所有项目通用 |
| 项目级 | `./CLAUDE.md` 或 `./.claude/CLAUDE.md` | 团队共享的项目约定 |
| 项目本地 | `./CLAUDE.local.md` | 个人的项目偏好，加到 `.gitignore` |
| 子目录 | `./src/api/CLAUDE.md` | 特定目录的局部规则 |

加载时，Claude Code 从文件系统根目录一路向下走到你的工作目录，每一层的 CLAUDE.md 和 CLAUDE.local.md 都会被读取。同一目录内，CLAUDE.local.md 在 CLAUDE.md 之后加载，所以你的个人偏好是"最后读到的"。

子目录的 CLAUDE.md 不会在启动时加载——它们是**惰性加载**的，只有当 Claude 读取到那个目录下的文件时才会触发。这个设计很聪明：一个大型 monorepo 里可能有几十个子目录各自的 CLAUDE.md，如果全部加载会浪费大量上下文空间。

### 组织级托管策略

企业用户可以部署一个全公司统一的 CLAUDE.md，由 IT 部门通过 MDM 或配置管理工具分发到每台开发机器上。这个文件的优先级最高，用户无法通过任何设置排除它。

适合放进去的内容：代码风格和质量标准、数据合规提醒、安全策略。不适合放的内容：具体的技术强制规则（那些应该用 managed settings 的 `permissions.deny` 来实现，因为设置是强制执行的，而 CLAUDE.md 只是"行为引导"）。

### 路径限定规则

除了子目录 CLAUDE.md，还有一种更灵活的方式：用 `.claude/rules/` 目录下的 Markdown 文件配合 frontmatter 的 `paths` 字段来限定规则的作用范围。

```markdown
---
paths:
  - "src/api/**/*.ts"
---

# API 开发规则

- 所有 API 端点必须包含输入验证
- 使用标准错误响应格式 { code, message }
- 加上 OpenAPI 文档注释
```

没有 `paths` 字段的规则文件在启动时就会加载，和 CLAUDE.md 一样。有 `paths` 字段的只在 Claude 读取匹配文件时才触发。

支持 glob 模式和花括号展开：

```markdown
---
paths:
  - "src/**/*.{ts,tsx}"
  - "tests/**/*.test.ts"
---
```

项目结构大概是这样：

```
your-project/
├── .claude/
│   ├── CLAUDE.md           # 主项目规则
│   └── rules/
│       ├── code-style.md   # 代码风格（全局加载）
│       ├── testing.md      # 测试约定（匹配测试文件时加载）
│       └── security.md     # 安全规则（全局加载）
```

这种方式比在子目录下放 CLAUDE.md 更模块化，也更容易团队维护。

### @导入

CLAUDE.md 支持用 `@` 语法导入其他文件：

```markdown
参考 @README.md 了解项目概况
参考 @package.json 了解可用的 npm 命令

# 额外规则
- git 工作流 @docs/git-instructions.md
```

导入的文件会在启动时展开并加载到上下文里。相对路径相对于包含 `@` 的那个文件，不是工作目录。最多支持 5 层嵌套导入。

不过要注意：导入不能减少上下文用量——导入的内容是直接展开的，和写在 CLAUDE.md 里没区别。它的好处是组织性更好，不是省 token。

### CLAUDE.md 会随项目演化

一个实际的演化过程大概是这样：

**第一周：** 跑 `/init` 生成初始版本，里面有基本的构建命令和项目结构。

**第二周：** Claude 在 review 时建议用 `var` 声明变量，你纠正了它，然后加一条"始终使用 `const`/`let`，禁止 `var`"。

**第一个月：** 发现规则越来越多了，把 API 相关的规则拆到 `.claude/rules/api.md`，测试相关的拆到 `.claude/rules/testing.md`，主 CLAUDE.md 只保留全局约定。

**三个月后：** 新同事加入，发现有些规则已经过时（比如已经弃用的库的约定），做一次清理。CLAUDE.md 是活的文档，不是写完就不管的。

### compact 后规则会丢吗

项目根目录的 CLAUDE.md 不会丢——`/compact` 之后 Claude Code 会从磁盘重新读取并注入上下文。但子目录的 CLAUDE.md 和 `.claude/rules/` 里有 `paths` 的规则文件不会自动重新注入，得等 Claude 下次读取匹配文件时才会触发。

如果你发现某条规则在 compact 后不生效了，大概率是因为：那条规则只是在对话里口头说过，没有写进 CLAUDE.md。解决方法很简单——把重要的规则写到 CLAUDE.md 里，不要依赖对话历史。

## Auto Memory：Claude 主动记住的事

Auto Memory 是 Claude Code 的自动记忆系统（需要 v2.1.59+）。当你和 Claude Code 协作时，它会主动把值得记住的信息写入 `~/.claude/projects/<project>/memory/` 目录里，下次对话时自动加载。

这里的 `<project>` 路径是从 git 仓库推导的，所以同一个仓库的所有 worktree 和子目录共享一个 Memory 目录。

### 什么会被记住

Auto Memory 专门记录"不在代码里但需要跨对话保留"的信息：

**用户偏好（user 类型）：**
- 技术背景和熟悉程度
- 沟通风格偏好
- 习惯用哪些工具

**反馈记录（feedback 类型）：**
- 你纠正过的行为（"别自动帮我 commit"）
- 你认可的做法（"这种写法很好，以后都这样"）
- 这类记忆最重要，防止 Claude 犯同样的错误

**项目背景（project 类型）：**
- 当前在做什么功能
- 为什么做某个技术决策
- 截止日期和约束

**外部资源（reference 类型）：**
- Bug 在哪里追踪（[Linear](https://linear.app)、[Jira](https://www.atlassian.com/software/jira) 项目名）
- 监控看板地址
- 内部文档位置

### 什么不该放进 Memory

这一条很多人搞混。以下内容不适合存到 Memory 里：

- **代码模式和架构** → 直接从代码里就能看出来，不需要记
- **git 历史** → 用 `git log` 查就行
- **bug 修复方案** → 修复已经在代码里了
- **CLAUDE.md 里已经有的规则** → 重复记忆只会浪费上下文空间
- **临时性的任务细节** → 做完就没用了，别让 Claude 记住

原则很简单：Memory 存的应该是"从代码和配置文件里推导不出来的、但跨对话有价值的信息"。

### Memory 的文件结构

```
~/.claude/projects/<project>/memory/
├── MEMORY.md              # 索引文件（每次对话都加载）
├── debugging.md           # 调试经验
├── api-conventions.md     # API 设计决策
├── user_profile.md        # 用户偏好
└── feedback_code_style.md # 代码风格反馈
```

**MEMORY.md 是核心。** 它是整个 Memory 目录的索引，记录着其他文件的位置和概要。每次对话启动时，MEMORY.md 的前 200 行或 25KB（取先到的那个限制）会被加载到上下文里。超过这个阈值的内容不会在启动时加载。

所以 Claude 会自己维护 MEMORY.md 的简洁性——把详细的内容移到单独的主题文件里，MEMORY.md 只保留索引。主题文件（比如 `debugging.md`、`api-conventions.md`）不会在启动时加载，Claude 需要的时候会自己用文件读取工具去读。

每个 Memory 文件的格式：

```markdown
---
name: 代码风格反馈
description: 用户对代码风格的明确要求
type: feedback
---

不要为假设的未来需求写抽象层。

**Why:** 用户曾指出一个不必要的策略模式重构。
**How to apply:** 改动范围严格匹配任务，不做超出任务的重构。
```

Memory 文件应该按主题组织（"调试经验"、"API 约定"、"用户偏好"），而不是按时间组织（"第一周记忆"、"第二周记忆"）。按主题组织方便 Claude 按需查找，按时间组织没有任何好处。

### 主动触发记忆保存

Claude Code 通常会自动判断什么值得记忆——你会看到界面上出现"Writing memory"的提示。但你也可以明确告诉它：

```
记住：这个项目用 pnpm，不用 npm
记住：不要在没有明确要求的情况下修改测试文件
```

Claude 不会每次对话都保存记忆。它会自己判断这条信息在未来的对话中是否有用，有用才记。

### 用 /memory 管理记忆

`/memory` 是查看和管理记忆的入口命令。运行后会展示：

- 当前加载的所有 CLAUDE.md 文件及路径
- Auto Memory 的开关状态
- 打开 Memory 文件夹的链接
- 选中任意文件可以在编辑器中打开

如果 Claude 没有遵守某条你以为它知道的规则，跑一下 `/memory` 确认规则文件确实被加载了。如果文件不在列表里，Claude 就看不到它。

想关闭 Auto Memory 可以在 `/memory` 里切换开关，或者在项目设置里设置 `"autoMemoryEnabled": false`。

### 管理已有记忆

Memory 文件是普通 Markdown，可以直接编辑：

```bash
# 查看记忆索引
cat ~/.claude/projects/*/memory/MEMORY.md

# 删除过时记忆
rm ~/.claude/projects/*/memory/project_old_feature.md
# 然后手动从 MEMORY.md 删掉对应行
```

或者在对话里说：
```
删掉关于 auth 重构的记忆，那个功能已经上线了
更新项目状态：现在在做用户通知系统
```

### 记忆会过时

这一点很多人忽略——记忆是有保质期的。

特别是 project 类型，任务完成了就该删掉。一个半年前的"当前任务"记忆留着，下次对话时 Claude 可能还以为那件事没做完。我自己踩过这个坑，某次 Claude 一上来就问我上次那个需求做完了吗，当时已经不记得那是什么事了。

使用原则：
- Memory 里提到的文件路径，Claude 应该在引用之前确认文件还存在
- 记忆和当前代码有矛盾时，以代码为准
- project 类型记忆在任务完成后主动删除
- 定期检查 MEMORY.md，清理不再相关的条目

## 三种机制的分工

| 机制 | 谁写 | 写什么 | 加载时机 | 生命周期 |
|------|------|--------|---------|---------|
| CLAUDE.md | 你 | 项目约定、行为规则 | 启动时全部加载 | 与代码库同步 |
| Auto Memory | Claude | 用户偏好、反馈、背景 | 启动时加载索引 | 长期，手动维护 |
| Skills | 你 | 可重复执行的工作流 | 触发时才加载全文 | 与代码库同步 |

### 判断信息该放哪里

遇到一条新信息，用这个思路判断：

**这是一条规则还是一个学到的偏好？**
- 规则 → CLAUDE.md
- 学到的偏好 → Memory

**这是一个多步骤的流程吗？**
- 是 → Skill
- 不是 → CLAUDE.md 或 Memory

**这条信息能从代码里直接看出来吗？**
- 能 → 哪都不用放，Claude 自己会看
- 不能 → CLAUDE.md 或 Memory

**这条信息跨对话有价值吗？**
- 有 → Memory
- 没有（只和当前任务相关） → 不用存

具体的例子：

- "项目用 pnpm" → 放 CLAUDE.md（团队约定）
- "这个用户习惯用 pnpm，之前有过一次提示 npm 被纠正" → Memory（学到的偏好）
- "每次提交前运行 lint 并检查类型" → Skill（多步骤工作流）
- "src/utils/helpers.ts 里有个 formatDate 函数" → 不用存（看代码就知道）
- "上次那个 auth bug 是因为 token 过期逻辑写反了" → 不用存（修复已经在代码里了）
- "deploy 到 staging 需要先跑 `make build-staging` 再用 `kubectl apply`" → CLAUDE.md（不明显的命令）

### 什么不该存到任何地方

再强调一遍，这些信息不需要持久化：

- **代码模式** → Claude 每次对话都会重新读代码
- **git 历史** → Claude 会用 `git log` 查
- **调试过程的中间步骤** → 修复已经在代码里了
- **临时任务细节** → 做完就没意义了
- **CLAUDE.md 里已有的内容** → 不要重复存到 Memory 里

## Memory 改善行为的实际例子

说了这么多理论，看一个实际的前后对比。

**没有 Memory 的情况：**

```
你：帮我加一个新的 API 端点 /api/users/search
Claude：好的，我来创建这个端点。
（Claude 创建了一个用 express 风格写的端点，用了 var 声明变量，
  没有输入验证，直接返回了原始数据库结果）
你：不对，我们用的是 Fastify，不是 Express。而且要加验证，要用 const……
```

**有了 Memory 之后：**

之前的对话里你纠正过 Claude，它把这些记到了 Memory：
- feedback: "项目用 Fastify，不是 Express"
- feedback: "所有 API 端点必须包含 Zod 输入验证"
- user: "用户偏好 const/let，纠正过 var 的使用"

```
你：帮我加一个新的 API 端点 /api/users/search
Claude：好的，我用 Fastify 的路由格式创建，加上 Zod schema 做输入验证。
（Claude 直接用了正确的框架、正确的验证方式、正确的变量声明）
```

这就是 Memory 的价值——把"你教过 Claude 的事"保留下来，避免每次对话都重新教一遍。

## 实践建议

**新项目启动时：**

1. 跑 `/init` 生成基础 CLAUDE.md
2. 补充项目特有的约定和禁止项
3. 把个人偏好告诉 Claude，让它写入 Memory
4. 用 `/memory` 确认一切正确加载

**日常协作时：**

- 发现 Claude 做了你不满意的事，明确纠正，并告诉它记住
- 开始新功能时，告诉它背景（"这是为了满足合规要求，不是重构技术债"）
- 功能完成后，让它清理相关的 project 记忆
- 对话之间用 `/clear` 保持上下文干净
- 长对话中主动 `/compact` 并指定保留重点

**团队协作时：**

- CLAUDE.md 提交到 git，所有人共享规则
- CLAUDE.local.md 放个人偏好，不提交
- 大规模项目用 `.claude/rules/` 按模块拆分规则
- Skills 提交到 git，共享工作流

**定期维护：**

- 每隔一两周检查一下 Memory，清理过时条目
- CLAUDE.md 跟着项目演化，去掉不再适用的规则
- 如果发现 CLAUDE.md 超过 200 行，考虑拆分到 `.claude/rules/`

## 小结

持久化上下文是 Claude Code 从"一次性工具"变成"长期协作伙伴"的关键。CLAUDE.md 让它记住规则，Memory 让它记住你，Skills 让它记住怎么做事。加上会话管理（`/compact`、`/clear`、`claude -c`），你可以在单次对话内保持上下文高效，也可以跨对话延续工作进度。三层持久化加上会话管理，每次打开项目，Claude Code 就已经是一个了解这个代码库、了解你偏好的协作者。
