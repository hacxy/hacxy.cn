# 第一章：Claude Code 是什么

如果你用过 ChatGPT 或者其他 AI 聊天工具来写代码，你一定遇到过这种情况：把一段代码粘进去，让它帮你修，它给出了答案，但你不确定这个修改放到整个项目里是否正确；或者你描述了一个需求，它生成了代码，但和你的已有架构完全对不上。

这个问题其实挺根本的——AI 看不到你的项目，它只能根据你贴进去的那一小块内容猜。

[Claude Code](https://claude.ai/code) 解决的正是这个问题。

## 一句话描述

Claude Code 是 [Anthropic](https://anthropic.com) 官方开发的 AI 编程助手，以命令行工具（CLI）的形式运行在你的终端里。它能直接读取你的项目文件、执行命令、修改代码，并在真实的开发环境中完成任务——不是生成一段"示例代码"让你自己去套，而是直接在你的项目里做事。

## 和普通 AI 聊天有什么不同

普通 AI 聊天工具（包括网页版 Claude）的工作方式是"你说我答"：你把代码贴给它，它输出修改后的代码，但它看不到你的完整项目，不知道其他文件里有什么，也无法直接运行或修改文件。

Claude Code 的工作方式完全不同：

| 对比维度 | 普通 AI 聊天 | Claude Code |
|---------|------------|-------------|
| 代码访问 | 你粘贴什么它就看什么 | 直接读取整个项目 |
| 文件操作 | 输出建议，你手动修改 | 直接编辑文件 |
| 命令执行 | 无法执行 | 可运行测试、构建等命令 |
| 上下文范围 | 单次对话窗口 | 项目全局 + 持久记忆 |
| 工作方式 | 问答式 | 代理式（主动完成任务） |

这种差异意味着你可以对 Claude Code 说"帮我把登录流程重构成 JWT"，它会自己去找相关文件、理解当前架构、做出修改、运行测试验证，而不是给你一段通用的 JWT 示例代码。

## 代理式工作方式

上面表格里提到了"代理式"，这个概念对新手来说可能比较抽象，但它其实是 Claude Code 和其他 AI 工具最核心的区别，得展开说说。

传统的 AI 编程助手是"一问一答"模式：你给它输入，它给你输出，完事。Claude Code 不一样，它走的是一个循环：

```
思考 → 行动 → 观察 → 再思考 → 再行动 → ...
```

举个具体的例子。你跟 Claude Code 说："项目的测试跑不过，帮我修一下。"

它不会直接问你"测试报错是什么"，而是自己开始干活：

1. 先跑一遍测试命令（`npm test`），看看到底哪些测试失败了
2. 读失败测试对应的测试文件，理解测试在验证什么
3. 再去读被测试的源码文件，找到出问题的逻辑
4. 修改源码
5. 重新跑测试，确认修复是否生效
6. 如果还有失败，继续分析、修改、验证，直到全部通过

整个过程你不用介入，它自己在"思考—行动—观察"的循环里不断推进。你只需要在最后审查它的改动就行。

说白了，普通 AI 聊天像是你在问一个顾问"你觉得这个怎么修"，而 Claude Code 更像你把任务交给一个同事，他自己去查资料、写代码、跑测试，搞定了再来跟你汇报。

这种工作方式的好处是显而易见的：对于那些需要跨多个文件、涉及多个步骤的任务，你不用一步步喂信息给它，也不用在 AI 和编辑器之间来回切换。不过，这也意味着你给它的指令越清晰，它的行动方向就越准确——[第六章](./06-efficiency-and-token-tips)会详细讲怎么给好指令。

## Claude Code 能做什么

### 理解代码库

你可以直接问它关于项目的问题，它会自己去读相关文件：

- "这个项目的认证逻辑在哪里？"
- "为什么首页加载这么慢？"
- "这个函数被哪些地方调用？"
- "帮我梳理一下这个项目的整体架构"

### 修改代码

告诉它要做什么，它会跨文件完成修改：

- "帮我给这个 Express 项目加上 CORS 支持"
- "这段代码有个 bug，用户登录后 token 没有被保存，帮我修"
- "把这个组件拆成三个更小的组件"
- "给 `UserService` 里的每个公共方法加上错误处理"

### 执行命令

它可以运行你项目里的命令：

- 跑测试并根据失败结果修复代码
- 执行构建，排查报错
- 运行 lint，修复所有 warning

### Git 操作

- 分析 staged 区的变更，生成规范的提交信息
- 创建分支、提交、甚至开 PR
- 解决合并冲突

### 代码审查和安全检查

这个能力容易被忽略，但其实非常实用：

- "帮我 review 一下这个 PR 的改动"
- "检查一下这个模块有没有安全漏洞"
- "这段处理用户输入的代码有没有 XSS 风险？"

### 调试

- 从报错信息出发，追踪问题根源
- 分析测试失败的原因，给出修复方案
- 直接把错误日志丢给它："这个报错怎么回事？帮我修了"

### 连接外部工具

通过 MCP（Model Context Protocol），Claude Code 可以连接外部服务——读 Notion 文档、更新 Jira ticket、拉 Slack 消息、查数据库。不过这个属于进阶用法，[第三章](./03-core-features)和[第五章](./05-settings-and-hooks)会详细讲。

## 运行在哪里

Claude Code 不只有一种使用方式，你可以根据习惯选择最顺手的。

### 终端 CLI

这是最核心、功能最完整的方式。在项目目录下运行 `claude` 命令就能进入交互式对话。

平台支持很全：macOS、Linux 都是一等公民；Windows 上推荐安装 [Git for Windows](https://git-scm.com/downloads/win)，这样 Claude Code 会使用 Bash 作为 shell；如果没装 Git for Windows，它会自动回退到 PowerShell。WSL 当然也支持。

安装方式也有好几种：官方安装脚本（推荐，会自动更新）、Homebrew、WinGet，Linux 上还有 apt / dnf / apk 包管理器可选。

### VS Code 扩展

在 VS Code 的扩展商店搜 "Claude Code" 就能装。装完之后你会在编辑器里看到一个聊天面板，可以用 `@` 引用当前文件，修改会以内联 diff 的形式展示，接受或拒绝一目了然。还支持 Cursor 等 VS Code 的衍生编辑器。

### JetBrains 插件

IntelliJ IDEA、WebStorm、PyCharm 等所有 JetBrains IDE 都支持。从 JetBrains Marketplace 装 [Claude Code 插件](https://plugins.jetbrains.com/plugin/27310-claude-code-beta-)就行。插件会调用你本地装好的 Claude Code CLI，所以需要先装 CLI。

### 桌面应用

独立的桌面应用，macOS 和 Windows 都有。界面比终端更直观，可以同时开多个会话，还能调度定时任务。适合不太习惯终端的用户。

### Web 端

打开 [claude.ai/code](https://claude.ai/code) 就能用，不需要本地安装任何东西。适合临时用一下，或者在没装开发环境的机器上干活。它还有一个 Remote Control 功能：你可以在 Web 上开始一个任务，然后用 `claude --teleport` 把会话拉到本地终端继续。

## 需要什么账号

Claude Code 需要付费账号才能使用，不同方案的差异主要在用量上限和可用模型：

- **Pro（$20/月）**：大部分个人开发者的起点。可以在终端、Web、桌面应用里使用 Claude Code，默认用 Sonnet 模型。如果你一天只有一两轮比较密集的使用，Pro 够用了。
- **Max 5x（$100/月）**：5 倍于 Pro 的用量上限，可以用 Opus 模型。适合每天都在用 Claude Code 做多文件修改的开发者。
- **Max 20x（$200/月）**：20 倍用量，全模型访问。如果你是 Claude Code 重度用户，或者在跑多 agent 并行任务，选这个。
- **Team（$100/人/月起）**：给团队用的，有管理后台，可以混合分配不同级别的 seat。
- **Enterprise**：大公司用的，有 SSO、数据隔离、合规审计之类的企业级功能。

还有一条路：不买订阅，直接用 [Anthropic Console](https://console.anthropic.com) 的 API key，按 token 付费。适合用量不稳定、或者想精确控制成本的情况。新注册的 API 账号通常会给一些免费额度。

选哪个？如果你刚开始尝试，Pro 就够了，用量不够再升级。模型方面，Sonnet 处理日常开发任务速度快、效果好；Opus 在复杂推理和大范围重构上更强，但每次调用消耗的 token 也更多。

## Claude Code 做不了什么

说了这么多能做的，也得聊聊它的局限，免得期望太高摔得更疼。

**它会犯错，而且有时候错得很自信。** Claude Code 生成的代码不一定是对的。它可能引入新的 bug，可能误解你的意图，可能在修一个文件的时候搞坏另一个文件。所以每次它做完改动，一定要 review。好在它修改文件的时候会展示 diff，你可以一个个确认。

**复杂的架构决策还是得靠人。** "我们应该用微服务还是 monorepo？""数据库选 PostgreSQL 还是 MongoDB？"这类问题 Claude Code 可以给你分析，但最终决策得你来做。它更适合执行层面的任务，而不是战略层面的判断。

**上下文长了之后质量会下降。** 虽然 Claude 的上下文窗口很大，但对话轮次太多之后，它可能会忘记前面讨论过的东西，或者开始重复自己。如果发现它"变笨了"，用 `/clear` 清一下上下文重新开始就好。

**它不能替代你理解自己的代码。** 把 Claude Code 当成一个效率倍增器，而不是一个可以完全替代你思考的东西。如果你自己都不理解项目的逻辑，直接让 Claude Code 去改，出了问题你也很难排查。

**指令越模糊，结果越不可控。** "帮我优化一下代码"这种模糊的指令，它可能会做一些你不想要的"优化"——比如加一堆过度抽象的封装。给它具体的、可验证的目标，效果会好很多。比起"优化代码"，"把这个函数的查询从 N+1 改成批量查询"就好多了。

## 小结

Claude Code 不是一个更聪明的代码补全工具，也不是一个会写代码的聊天机器人。它更接近于一个能独立工作的初级工程师——你告诉它目标，它自己去探索、去修改、去验证，最后把结果交给你审查。

理解这一点，是用好 Claude Code 的前提。
