# Claude Code Memory：让 AI 记住你的一切

每次打开新对话，Claude 都是"失忆"状态——不知道你习惯用 pnpm 还是 npm，不记得上周你说过不要在注释里写"what"只写"why"，也不知道这个项目正在经历什么。

Memory 系统就是为了解决这件事：把跨对话需要保留的信息持久化到文件，让 Claude 在未来的每次对话中都能带着这些背景上下文工作。

## Memory 的本质

Memory 是一套约定好的文件目录，放在 `.claude/projects/<project-id>/memory/` 下。每条记忆是一个独立的 Markdown 文件，有固定的 frontmatter 结构；`MEMORY.md` 是索引文件，列出所有记忆条目的简短描述和链接。

每次对话开始时，Claude 会加载 `MEMORY.md`，根据对话内容判断哪些记忆文件与当前任务相关，再按需读取。

这个设计的优势在于：**记忆是可审查的、可版本管理的、可手动编辑的**。你随时可以打开文件夹看看 Claude 记住了什么，也可以直接修改或删除不再准确的条目。

## 四种记忆类型

Memory 按内容性质分为四类，对应不同的使用场景。

### user — 关于你是谁

记录用户的角色、背景知识、偏好和工作方式。目标是让 Claude 知道"我在和什么样的人合作"，从而调整表达方式和建议深度。

**典型内容：**
- 技术栈熟悉程度（"Go 后端十年经验，React 刚入门"）
- 职责方向（"数据科学家，关注可观测性"）
- 沟通风格偏好（"不要废话，直接给结论"）

**示例文件 `memory/user_profile.md`：**

```markdown
---
name: 用户技术背景
description: 用户的技术方向、熟悉程度和工作重点
type: user
---

主要方向是前端工程化，熟悉 TypeScript、React、Vite 工具链。
对 Node.js 后端有基础了解，不熟悉运维和 DevOps。
喜欢简洁直接的代码，不需要过多注释和冗余封装。
```

### feedback — 你教过我的事

记录用户纠正过的行为和认可的方式，是最重要的记忆类型。核心原则是：**不要让用户说第二遍相同的话**。

不只记录"不要这样做"的纠正，也要记录"这样做是对的"的认可——如果只保存错误反馈，会让 Claude 变得过于保守。

**典型内容：**
- 代码风格要求（"不要用 any，哪怕临时图省事"）
- 工作流偏好（"不要在我没有让你 commit 的时候自动提交"）
- 表达方式要求（"回复不要超过 3 句话，我能看 diff"）

**示例文件 `memory/feedback_code_style.md`：**

```markdown
---
name: 代码风格反馈
description: 用户对代码风格和实现方式的明确要求
type: feedback
---

不要为假设的未来需求写抽象层。三行相似的代码好过一个过早的封装。

**Why:** 用户曾经指出我把一个简单的条件判断改成了策略模式，实际没必要。
**How to apply:** 改动范围严格匹配任务要求，不做超出任务的重构或设计。
```

### project — 当前在做什么

记录项目的进行中状态、决策背景、截止时间等。这类记忆衰减快，需要及时更新。

**典型内容：**
- 正在进行的功能或重构方向
- 为什么做某个技术选型（不是选了什么，而是为什么）
- 阶段性约束（冻结发布、上线前后）

**示例文件 `memory/project_auth_rewrite.md`：**

```markdown
---
name: 认证模块重构背景
description: 当前认证模块重写的动机和约束
type: project
---

正在重写旧的 auth 中间件，预计 2026-05-20 完成。

**Why:** 法务部门要求 session token 的存储方式必须符合新的合规标准，不是技术债清理。
**How to apply:** 相关 PR 的范围决策优先满足合规要求，而不是顺手优化其他东西。
```

### reference — 去哪里找信息

记录外部系统的地址和用途，让 Claude 知道"去哪里查"。

**典型内容：**
- Bug 追踪系统（Linear、Jira 项目名）
- 监控看板地址
- 内部文档位置

**示例文件 `memory/reference_systems.md`：**

```markdown
---
name: 外部系统参考
description: 项目相关的外部工具和资源地址
type: reference
---

- Pipeline 相关 bug 追踪在 Linear 项目 "INGEST"
- API 延迟监控看板：grafana.internal/d/api-latency（oncall 盯这个）
- 内部组件库文档：design.internal/components
```

## MEMORY.md：记忆索引

所有记忆文件都需要在 `MEMORY.md` 里登记一行，格式是：

```markdown
- [记忆名称](file.md) — 一句话说明内容
```

`MEMORY.md` 是每次对话都会加载的索引，**超过 200 行会被截断**，所以要保持精简，每行控制在 150 字以内。

一个典型的 `MEMORY.md`：

```markdown
- [用户技术背景](user_profile.md) — 前端工程化方向，TypeScript/React 熟练，不熟悉运维
- [代码风格反馈](feedback_code_style.md) — 不做超出任务的抽象，不用 any
- [提交方式反馈](feedback_commit.md) — 只在明确要求时才 commit，不自动提交
- [认证重构背景](project_auth_rewrite.md) — 合规驱动的 auth 重写，截止 2026-05-20
- [外部系统参考](reference_systems.md) — Linear/Grafana/内部文档地址
```

## 什么时候保存记忆

**应该保存：**
- 用户第一次告诉你某个偏好或纠正了某个行为
- 你了解到项目背景中不能从代码推断的信息（为什么做，不只是做了什么）
- 用户明确说"记住"或"下次也这样"

**不应该保存：**
- 能从代码直接读到的架构信息（文件结构、函数签名）
- git 历史（`git log` 是权威来源）
- 当前对话的临时状态（用 TodoWrite 管理）
- 已经写在 CLAUDE.md 里的内容

## 记忆的生命周期

记忆会过时。一个三个月前写的 project 记忆，描述的项目可能早已上线；一个 user 记忆说"刚开始学 React"，现在可能已经是熟练用户了。

几个处理原则：

**验证后再用。** 如果记忆里提到某个文件路径或函数名，在引用前先确认它仍然存在。"记忆说 X 存在"不等于"X 现在存在"。

**发现矛盾时，信眼前的代码。** 如果当前读到的代码与记忆描述不符，以代码为准，然后更新或删除过时的记忆。

**主动清理。** Project 类记忆尤其需要清理，任务完成后及时删除，避免过时信息干扰判断。

## 和其他机制的配合

Claude Code 有几套持久化机制，分工不同：

| 机制 | 用途 | 生命周期 |
|------|------|----------|
| Memory | 跨对话的用户偏好、项目背景 | 长期，手动维护 |
| CLAUDE.md | 项目约定、架构文档、开发规范 | 与代码库同步 |
| Skills | 可重复执行的工作流（提交、写博客） | 长期，随项目管理 |
| Todo/Tasks | 当前对话的任务拆解和进度 | 单次对话 |

Memory 不是用来记录"这个函数怎么写"的——那是 CLAUDE.md 的职责。Memory 是用来记录"这个人是谁、他们在做什么、他们希望你怎么工作"。

## 实际工作流

### 第一次建立记忆

打开新项目后，和 Claude 对话几轮，逐渐告诉它背景信息。每次你说"我们用 pnpm"、"这个项目是给内部工具的"、"代码里不要加 TODO 注释"，Claude 应该把这些信息保存下来。

如果 Claude 没有主动保存，可以直接说："记住这个偏好。"

### 验证记忆是否生效

在新的对话开始时，你可以问 Claude："你知道这个项目在做什么吗？"或者"你记得我的代码风格要求吗？"来检验记忆是否被正确加载。

### 手动管理记忆

记忆文件是普通 Markdown，可以直接编辑：

```bash
# 查看所有记忆
cat .claude/projects/.../memory/MEMORY.md

# 删除过时记忆
rm .claude/projects/.../memory/project_old_feature.md
# 然后手动从 MEMORY.md 删掉对应行
```

或者直接告诉 Claude："删掉关于 X 的记忆"、"更新一下项目背景"。

## 小结

Memory 系统的价值在于把隐性的协作知识变成显性的文件。好的记忆让 Claude 第一次对话就像一个熟悉你工作方式的同事；没有记忆，每次对话都要重新建立上下文。

建立记忆的成本很低，维护记忆的关键是及时清理——特别是 project 类型的，完成的事情就删掉，让记忆库保持轻量和准确。
